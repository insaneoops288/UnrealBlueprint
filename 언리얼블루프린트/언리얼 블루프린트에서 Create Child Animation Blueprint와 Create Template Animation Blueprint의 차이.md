# **자식 vs 템플릿 애니메이션 블루프린트 비교**

애니메이션 시스템을 설계할 때, 중복되는 로직을 줄이기 위해 아래 두 가지 방식을 선택할 수 있습니다.

## **1\. 자식 애니메이션 블루프린트 (Child Animation Blueprint)**

전형적인 객체 지향 프로그래밍의 '상속' 개념입니다.

* **스켈레톤 종속성**: 부모와 자식이 **동일한 스켈레톤**을 공유해야 합니다. (혹은 호환 가능한 스켈레톤)  
* **주요 목적**: 동일한 로직과 그래프 구조를 유지하면서, 특정 애니메이션 에셋(시퀀스, 블렌드 스페이스 등)만 교체하고 싶을 때 사용합니다.  
* **작동 방식**:  
  * 부모의 \*\*이벤트 그래프(Event Graph)\*\*와 **애님 그래프(Anim Graph)** 구조를 그대로 물려받습니다.  
  * 자식 블루프린트의 **에셋 오버라이드(Asset Override)** 탭에서 부모가 사용하던 애니메이션을 자신의 스켈레톤에 맞는 애니메이션으로 교체합니다.  
* **예시**: '인간'이라는 부모 아래 '걷기 속도가 다른 인간'이나 '약간의 변형된 포즈를 가진 인간'을 만들 때 적합합니다.

## **2\. 템플릿 애니메이션 블루프린트 (Template Animation Blueprint)**

UE5에서 도입된 기능으로, 스켈레톤에 구애받지 않는 "로직 설계도"입니다.

* **스켈레톤 종속성**: 생성 시점에 **스켈레톤을 지정하지 않습니다.** 즉, 스켈레톤 독립적(Skeleton-agnostic)입니다.  
* **주요 목적**: 서로 **다른 스켈레톤**을 가진 여러 캐릭터들이 **동일한 애니메이션 로직**(예: 특정 상태 머신, 복잡한 레이어 구조)을 공유해야 할 때 사용합니다.  
* **작동 방식**:  
  * 템플릿에서 공통 로직을 작성합니다. 이때 애니메이션 에셋은 '가상의 데이터'처럼 취급됩니다.  
  * 실제 캐릭터에 적용할 때는 이 템플릿을 기반으로 새로운 애니메이션 블루프린트를 생성하며, 이때 비로소 특정 스켈레톤을 지정합니다.  
  * 지정된 스켈레톤에 맞춰 애니메이션 에셋들을 채워 넣습니다.  
* **예시**: '2족 보행 몬스터'라는 템플릿을 하나 만들고, 이를 통해 '오크', '고블린', '트롤' 등 골격 구조가 전혀 다른 캐릭터들에게 동일한 AI 애니메이션 로직을 적용할 때 강력합니다.

## **요약 비교표**

| 구분 | 자식 애니메이션 블루프린트 | 템플릿 애니메이션 블루프린트 |
| :---- | :---- | :---- |
| **스켈레톤** | 부모와 동일해야 함 | 생성 시 필요 없음 (독립적) |
| **재사용 범위** | 동일 스켈레톤 내 변형 | 서로 다른 스켈레톤 간 로직 공유 |
| **핵심 기능** | 에셋 오버라이드 (Asset Override) | 로직 템플릿화 (Logic Templating) |
| **사용 시점** | 캐릭터 외형이나 애니메이션만 살짝 바뀔 때 | 복잡한 애님 그래프 구조를 여러 종족에 쓸 때 |

### **결론**

* \*\*"똑같은 뼈대를 가진 캐릭터인데 애니메이션 파일만 바꾸고 싶다"\*\*면 **자식 블루프린트**를 사용하세요.  
* \*\*"뼈대는 다르지만 동작하는 논리(상태 머신 등)는 똑같이 쓰고 싶다"\*\*면 **템플릿 블루프린트**가 정답입니다.
