# **Get All Actors Of Class 노드 완전 정복**

Get All Actors Of Class 노드는 레벨에 배치된 특정 종류의 액터들을 한꺼번에 불러와서 무언가 작업을 하고 싶을 때 사용합니다.

## **1\. 주요 역할**

* **액터 검색**: 월드 내에 스폰되어 있는 특정 클래스(예: BP\_Enemy, BP\_Item 등)의 모든 인스턴스를 찾습니다.  
* **배열 생성**: 찾은 액터들을 하나의 리스트(배열)로 묶어서 내보냅니다.  
* **일괄 제어**: 배열에 담긴 액터들을 ForEachLoop 노드와 연결하여 한꺼번에 투명하게 만들거나, 파괴하거나, 특정 변수 값을 수정하는 등의 작업을 할 수 있습니다.

## **2\. 노드 파라미터 (Input/Output)**

* **Actor Class (Input)**: 찾고자 하는 대상 클래스를 선택합니다. (자식 클래스도 포함해서 검색됩니다.)  
* **Out Actors (Output)**: 검색된 액터들이 담긴 배열입니다. 이 출력 핀을 드래그해서 ForEachLoop나 Get (Copy) 노드에 연결합니다.

## **3\. 대표적인 활용 사례**

* **게임 종료 조건 체크**: 레벨에 남은 적(BP\_Enemy)의 숫자가 0인지 확인할 때.  
* **환경 설정**: 게임 시작 시 레벨에 배치된 모든 전등(BP\_Light)의 색상을 한꺼번에 바꿀 때.  
* **특정 액터 참조**: 레벨에 딱 하나만 존재하는 매니저 액터나 플레이어 점수판 액터의 참조를 가져오고 싶을 때 (인덱스 0번 사용).

## **4\. ⚠️ 성능 주의사항 (매우 중요)**

이 노드는 사용하기 편리하지만, 남용하면 게임 성능(프레임 드랍)에 큰 영향을 줄 수 있습니다.

1. **무거운 연산**: 레벨에 존재하는 모든 액터를 전수 조사하기 때문에 비용이 큽니다.  
2. **Tick 금지**: 절대 **Event Tick** (매 프레임 호출)에서 사용하지 마세요.  
3. **캐싱(Caching) 권장**: BeginPlay에서 한 번 실행해서 변수에 저장(Promote to Variable)해두고 재사용하는 것이 좋습니다.  
4. **대안 고려**:  
   * 액터가 생성될 때 특정 '게임 상태(GameState)'나 '매니저'에 자신을 등록하게 만드는 방식이 훨씬 효율적입니다.  
   * 특정 그룹을 찾을 때는 **태그(Tag)** 시스템을 사용하는 Get All Actors with Tag도 고려해 보세요.

## **5\. 요약**

| 장점 | 단점 |
| :---- | :---- |
| 구현이 매우 쉽고 직관적임 | 성능 소모가 큼 (O(n) 검색) |
| 클래스 단위로 깔끔하게 필터링됨 | 액터 수가 많을수록 로딩/연산 시간이 길어짐 |
| 별도의 참조 설정 없이도 통신 가능 | 실행 시점(Timing)에 따라 검색 결과가 달라질 수 있음 |
