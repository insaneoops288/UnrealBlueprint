# **언리얼 엔진 IsServer 노드 상세 가이드**

언리얼 엔진의 멀티플레이어 환경에서 **IsServer** 노드는 현재 해당 블루프린트 로직이 실행되고 있는 '인스턴스'가 서버인지 클라이언트인지를 판별하는 가장 기본적인 도구입니다.

## **1\. IsServer 노드란?**

IsServer 노드는 불리언(Boolean) 값을 반환하는 순수 함수(Pure Function)입니다.

* **True 반환:** 현재 게임 인스턴스가 **데디케이트 서버(Dedicated Server)** 또는 \*\*리슨 서버(Listen Server, 방장의 PC)\*\*일 때.  
* **False 반환:** 현재 게임 인스턴스가 \*\*클라이언트(접속한 유저의 PC)\*\*일 때.

## **2\. 주요 역할과 필요성**

멀티플레이어 게임에서는 동일한 코드가 서버와 여러 클라이언트에서 동시에 실행될 수 있습니다. 이때 IsServer는 다음과 같은 역할을 수행합니다.

### **① 서버 전용 로직의 보호 (Authority)**

게임의 점수 계산, 아이템 획득, 데미지 처리 등 게임의 '진실'을 결정하는 로직은 반드시 서버에서만 처리되어야 합니다. 클라이언트에서 이를 처리하면 해킹(변조)의 위험이 있기 때문입니다.

### **② 중복 실행 방지**

예를 들어, 몬스터를 스폰하는 로직이 서버와 클라이언트 모두에서 실행되면, 서버에 1마리, 클라이언트에 각각 1마리씩 중복 생성되는 문제가 발생합니다. IsServer를 통해 서버에서만 스폰하게 하면 네트워크 복제(Replication)를 통해 모든 클라이언트에 정상적으로 동기화됩니다.

### **③ 리소스 최적화**

서버는 화면을 그릴 필요가 없는 경우가 많습니다. 반대로 클라이언트는 복잡한 연산보다는 UI 연출이나 사운드 재생이 중요합니다. IsServer를 사용해 "서버는 연산만, 클라이언트는 연출만" 하도록 분기할 수 있습니다.

## **3\. IsServer vs HasAuthority**

많은 입문자가 HasAuthority 노드와 헷갈려 합니다. 차이점은 다음과 같습니다.

| 구분 | IsServer | HasAuthority |
| :---- | :---- | :---- |
| **판단 기준** | **게임 인스턴스** 그 자체 (서버인가?) | **액터의 소유권** (이 액터를 제어할 권한이 있는가?) |
| **클라이언트에서의 값** | 항상 **False** | 소유 중인 액터(예: 자신의 캐릭터)에 대해서는 **True**일 수 있음 |
| **주된 용도** | 서버 전용 시스템 로직 실행 | 특정 액터의 상태 변경 권한 확인 |

**팁:** 단순히 "이 코드가 서버 PC에서 돌아가야 하는가?"를 묻는다면 IsServer가 더 직관적입니다.

## **4\. 실전 활용 예시 (Blueprints)**

### **예시 A: 데미지 처리 (서버 전용)**

1. AnyDamage 이벤트 발생.  
2. IsServer 노드로 분기(Branch).  
3. **True**일 때만 체력 변수를 감소시킴. (클라이언트에서 체력을 깎아봤자 서버에 반영되지 않으므로 서버에서 처리)

### **예시 B: UI 표시 (클라이언트 전용)**

1. 아이템 습득 이벤트 발생.  
2. IsServer 노드로 분기.  
3. **False**일 때만 "아이템을 획득했습니다\!"라는 위젯을 화면에 띄움. (서버는 화면이 없으므로 UI를 띄울 필요가 없음)

## **5\. 주의사항**

* **Dedicated Server 체크:** 리슨 서버 환경(방장이 직접 플레이하는 경우)에서는 IsServer가 True이면서 동시에 로컬 플레이어이기도 합니다. 이를 고려하지 않으면 방장에게만 유리한 로직이 생길 수 있습니다.  
* **네트워크 복제와의 연계:** IsServer로 서버에서 값을 바꿨다면, 해당 변수가 Replicated 설정이 되어 있어야 클라이언트들도 그 변화를 알 수 있습니다.
