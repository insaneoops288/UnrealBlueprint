# **언리얼 블루프린트: TMap Find 노드 가이드**

언리얼 엔진의 TMap 데이터 구조에서 **Find** 노드는 특정 항목에 접근하기 위한 가장 기본적인 도구입니다.

## **1\. Find 노드의 주요 역할**

Find 노드는 맵에 저장된 데이터 중 사용자가 제공한 \*\*키(Key)\*\*와 일치하는 항목을 검색하여, 그에 대응하는 \*\*값(Value)\*\*을 반환합니다.

## **2\. 입력 및 출력 핀 구성**

### **입력 (Inputs)**

* **Target Map**: 검색 대상이 될 TMap 변수입니다.  
* **Key**: 찾고자 하는 특정 키 값입니다. (예: 아이템 이름, 플레이어 ID 등)

### **출력 (Outputs)**

* **Value**: 입력한 키와 연결된 실제 데이터 값입니다.  
* **Return Value (Boolean)**: 검색 성공 여부를 알려줍니다.  
  * True: 맵 안에 해당 키가 존재하며, 값을 성공적으로 찾음.  
  * False: 맵 안에 해당 키가 존재하지 않음.

## **3\. 작동 원리 및 특징**

### **성공 여부 확인 (Branch 필수)**

Find 노드를 사용할 때는 단순히 값만 가져오는 것이 아니라, \*\*Boolean 출력값(Return Value)\*\*을 **Branch** 노드에 연결하여 데이터가 실제로 존재하는지 먼저 확인하는 것이 블루프린트 프로그래밍의 표준 관례입니다.

### **데이터 안전성**

만약 존재하지 않는 키를 입력했는데도 성공 여부를 확인하지 않고 출력된 Value를 그대로 사용하려 하면, 해당 값의 타입에 따른 기본값(예: 숫자는 0, 객체는 Null)이 나오거나 예상치 못한 논리적 오류가 발생할 수 있습니다.

### **성능 (Time Complexity)**

TMap은 내부적으로 해시 테이블(Hash Table)을 사용하므로, Find 노드의 탐색 속도는 매우 빠릅니다. 데이터의 양이 많아져도 검색 속도가 거의 일정하게 유지되는 $O(1)$의 성능을 보여줍니다.

## **4\. 활용 예시: 인벤토리 시스템**

예를 들어 TMap\<FName, int32\> 타입의 인벤토리(아이템 이름 : 수량)가 있다고 가정해 봅시다.

1. **Find** 노드에 'Portion'이라는 키를 입력합니다.  
2. **Return Value**가 True라면 \-\> **Branch**의 True 실행 핀을 타고 아이템 개수(Value)를 UI에 표시합니다.  
3. **Return Value**가 False라면 \-\> **Branch**의 False 실행 핀을 타고 "아이템이 없습니다"라는 메시지를 출력합니다.

## **5\. Find vs Contains 노드 차이점**

| 비교 항목 | Find 노드 | Contains 노드 |
| :---- | :---- | :---- |
| **목적** | 값을 가져오기 위해 사용 | 존재 여부만 확인하기 위해 사용 |
| **반환값** | 데이터 값(Value) \+ 성공 여부(Bool) | 성공 여부(Bool)만 반환 |
| **용도** | 해당 데이터를 활용해야 할 때 | 단순히 들어있는지만 체크할 때 |

## **요약**

**Find** 노드는 "이 키에 해당하는 데이터가 있어? 있다면 그 값을 나한테 줘"라고 묻는 노드입니다. 항상 **Branch** 노드와 함께 사용하여 데이터의 유효성을 검사하는 습관을 들이는 것이 중요합니다.
