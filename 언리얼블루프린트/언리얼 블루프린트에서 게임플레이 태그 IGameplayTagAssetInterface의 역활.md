# **IGameplayTagAssetInterface의 역할과 상세 가이드**

언리얼 엔진(Unreal Engine) 개발, 특히 \*\*게임플레이 어빌리티 시스템(GAS)\*\*을 사용하거나 복잡한 상태 기반 로직을 설계할 때 IGameplayTagAssetInterface는 객체 간의 통신을 추상화하는 핵심적인 역할을 합니다.

## **1\. 핵심 역할: "태그 소유의 표준화"**

가장 큰 역할은 \*\*"이 객체가 어떤 태그를 가지고 있는지 묻는 공통된 언어"\*\*를 제공하는 것입니다.

* **상태 노출:** 특정 액터나 데이터 에셋이 '독에 걸림(State.Poisoned)', '비행 중(State.Flying)'과 같은 상태 태그를 가지고 있을 때, 외부에서 이를 일관된 방식으로 확인하게 해줍니다.  
* **형 변환(Casting) 최소화:** 상대방이 정확히 어떤 클래스인지 몰라도, 이 인터페이스를 구현하고 있다는 것만 알면 바로 태그를 쿼리할 수 있습니다.  
* **GAS와의 연동:** Gameplay Ability System은 이 인터페이스를 통해 어빌리티 실행 가능 여부나 이펙트 중첩 등을 판단합니다.

## **2\. 주요 함수: GetOwnedGameplayTags**

이 인터페이스를 구현할 때 가장 핵심이 되는 함수는 C++ 기준 GetOwnedGameplayTags입니다.

* **기능:** 해당 객체가 현재 소유한 모든 태그를 FGameplayTagContainer에 담아서 반환합니다.  
* **블루프린트에서의 모습:** 인터페이스를 추가하면 'Interfaces' 섹션에 이 함수가 나타나며, 여기서 해당 액터의 태그 변수를 리턴하도록 설정합니다.

## **3\. 왜 사용하는가? (이점)**

### **① 디커플링 (Decoupling)**

만약 인터페이스가 없다면, A\_Character의 태그를 검사하기 위해 A\_Character로 캐스팅해야 하고, B\_Prop을 검사하려면 다시 B\_Prop으로 캐스팅해야 합니다. 인터페이스를 사용하면 "인터페이스를 구현했는가?"만 확인하면 끝납니다.

### **② 게임플레이 태그 쿼리 (Tag Queries)**

단순히 "태그가 있는가?"를 넘어, "태그 A는 있고 B는 없는가?"와 같은 복잡한 조건식(Gameplay Tag Query)을 실행할 때 인터페이스를 통해 객체의 태그 컨테이너를 즉시 가져올 수 있습니다.

### **③ 유연한 설계**

액터뿐만 아니라 PrimaryDataAsset 같은 데이터 전용 객체에도 이 인터페이스를 구현하여, 특정 아이템이 어떤 속성(태그)을 가졌는지 일관되게 관리할 수 있습니다.

## **4\. 블루프린트 구현 방법**

1. **Class Settings:** 블루프린트 상단의 'Class Settings' 클릭.  
2. **Interfaces:** 'Implemented Interfaces' 항목에서 GameplayTagAssetInterface를 추가.  
3. **Variable 생성:** 해당 블루프린트에 Gameplay Tag Container 타입의 변수를 생성 (예: MyTags).  
4. **Interface 구현:** 내비게이터의 Interfaces 항목에서 GetOwnedGameplayTags를 더블 클릭하여 열고, 생성한 MyTags 변수를 출력(Output)에 연결.

## **5\. 실무 활용 예시**

* **데미지 시스템:** 화염 데미지를 줄 때, 타겟이 Resistance.Fire 태그를 가지고 있는지 인터페이스를 통해 확인하여 데미지를 경감시킵니다.  
* **UI 업데이트:** 현재 플레이어가 소유한 태그 리스트를 인터페이스로 가져와서 상태 이상 아이콘을 화면에 표시합니다.  
* **상호작용:** 문(Door) 액터가 Lock.Key\_Blue 태그를 요구할 때, 플레이어 인벤토리(인터페이스 구현체)에 해당 태그가 있는지 체크합니다.

## **요약**

IGameplayTagAssetInterface는 \*\*"너 무슨 태그 가지고 있어?"\*\*라는 질문에 답할 수 있는 능력을 객체에게 부여하는 것입니다. 이를 통해 코드의 의존성을 줄이고 가독성과 확장성이 높은 게임 로직을 짤 수 있게 됩니다.
