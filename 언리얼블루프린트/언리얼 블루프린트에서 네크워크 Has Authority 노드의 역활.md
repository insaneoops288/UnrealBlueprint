# **언리얼 엔진 블루프린트: Has Authority 노드 완벽 이해**

멀티플레이어 게임 제작 시, 특정 로직이 **서버**에서 실행되어야 할지 **클라이언트**에서 실행되어야 할지를 결정하는 것은 매우 중요합니다. Has Authority 노드는 바로 이 결정을 돕는 체크포인트 역할을 합니다.

## **1\. 노드의 기본 개념**

Has Authority 노드는 실행 핀을 받아 현재 액터의 **네트워크 권한(Network Authority)** 상태를 확인하고, 두 가지 경로로 분기합니다.

* **Authority (권한 있음):** 현재 로직이 \*\*서버(Server)\*\*에서 실행 중임을 의미합니다. (싱글 플레이어 게임의 경우 항상 이 경로로 실행됩니다.)  
* **Remote (원격):** 현재 로직이 \*\*클라이언트(Client)\*\*에서 실행 중임을 의미합니다.

## **2\. 왜 사용하는가? (역할)**

### **서버 측 로직 보호 (Anti-Cheat)**

중요한 게임 데이터(체력 감소, 아이템 획득, 점수 계산 등)는 오직 서버에서만 처리되어야 합니다. 클라이언트가 자신의 체력을 마음대로 수정하는 것을 방지하기 위해 Has Authority를 사용하여 서버 경로에서만 해당 수치를 변경하도록 제한합니다.

### **중복 실행 방지**

예를 들어, 서버와 클라이언트 모두에서 액터가 스폰(Spawn)되는 로직이 있다면, Has Authority 체크 없이 실행할 경우 서버에서도 스폰하고 클라이언트에서도 개별적으로 스폰하여 월드에 중복된 객체가 생길 수 있습니다.

### **역할 분담**

* **서버:** 게임의 규칙 관리, 물리 연산의 최종 결정, 데이터베이스 통신.  
* **클라이언트:** 시각적 효과(FX), 사운드 재생, UI 업데이트, 로컬 입력 처리.

## **3\. 실무 활용 예시**

### **예시 A: 체력 감소 로직 (데미지 처리)**

데미지는 게임의 결과에 직접적인 영향을 주므로 반드시 서버에서 처리해야 합니다.

1. Event Any Damage 발생.  
2. Has Authority 노드 연결.  
3. **Authority** 핀에만 Health 변수 차감 로직 연결.  
4. (선택사항) **Remote** 핀에는 클라이언트가 데미지를 입었을 때 화면이 붉어지는 UI 효과 연결.

### **예시 B: 아이템 습득**

1. 캐릭터가 아이템에 오버랩(Overlap)됨.  
2. Has Authority 체크.  
3. **Authority**: 서버 인벤토리에 아이템 추가 및 월드에서 아이템 제거(Destroy).  
4. **Remote**: 아무것도 하지 않음 (서버에서 아이템이 제거되면 리플리케이션에 의해 클라이언트에서도 사라짐).

## **4\. 네트워크 역할(Role) 비교**

이 노드는 내부적으로 액터의 Role 변수를 확인합니다.

| 상태 | Role 값 | Has Authority 결과 |
| :---- | :---- | :---- |
| **서버(세션 호스트)** | Role\_Authority | **Authority** |
| **클라이언트** | Role\_SimulatedProxy 또는 Role\_AutonomousProxy | **Remote** |

## **5\. 주의사항**

* **리플리케이션(Replication)과의 관계:** 변수가 리플리케이트(RepNotify 등) 설정되어 있다면, 서버에서 값을 바꾸기만 하면 클라이언트로 자동 전달됩니다. 따라서 Has Authority의 **Authority** 섹션에서만 값을 수정하는 것이 정석입니다.  
* **컴포넌트에서의 사용:** 컴포넌트 내부에서도 사용할 수 있으며, 해당 컴포넌트를 소유한 액터의 권한 상태를 따릅니다.

**요약하자면:** Has Authority는 \*\*"내가 이 게임의 주인(서버)인가?"\*\*를 묻는 질문이며, 게임의 무결성을 유지하고 서버-클라이언트 간의 역할을 분리하는 핵심 장치입니다.
