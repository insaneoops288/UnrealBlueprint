# **Unreal Engine: Switch has Authority 노드 이해하기**

멀티플레이어 게임 제작 시, 어떤 로직은 **서버**에서만 실행되어야 하고(예: 데미지 처리, 아이템 획득), 어떤 로직은 **클라이언트**에서도 실행되어야 합니다(예: UI 업데이트, 사운드 재생). 이를 제어하는 가장 기본적인 노드가 바로 Switch has Authority입니다.

## **1\. 노드의 구조**

이 노드는 실행 핀을 두 가지 경로로 분기합니다.

* **Authority (권한 있음):** 코드를 실행 중인 머신이 해당 액터에 대해 '주도권'을 가지고 있을 때 실행됩니다. 보통 **서버(Server)** 혹은 리슨 서버의 호스트가 이 경로를 탑니다.  
* **Remote (원격):** 코드를 실행 중인 머신이 해당 액터의 복제본을 가지고 있는 \*\*클라이언트(Client)\*\*일 때 실행됩니다.

## **2\. 주요 역할**

### **① 보안 및 데이터 무결성 유지 (Anti-Cheat)**

게임의 중요한 데이터(체력, 점수, 인벤토리)는 클라이언트가 마음대로 수정하게 두어서는 안 됩니다.

* **Authority 경로:** 서버에서만 중요한 수치를 변경하도록 설계하여 해킹이나 변조를 방지합니다.

### **② 중복 실행 방지**

만약 "아이템 상자를 열었을 때 아이템을 생성하는 로직"에 이 노드가 없다면, 서버와 모든 클라이언트에서 각각 아이템이 생성되어 게임 엔진에 과부하가 걸리거나 데이터가 꼬이게 됩니다.

* 서버(Authority)에서만 아이템을 생성(Spawn Actor)하고, 클라이언트들에게는 생성된 사실을 \*\*리플리케이션(Replication)\*\*을 통해 전달해야 합니다.

### **③ 시각적 효과 분리**

* **Remote 경로:** 서버는 굳이 계산할 필요가 없는 클라이언트 전용 연출(화면 흔들림, 로컬 UI 팝업, 개인용 파티클 등)을 처리할 때 사용합니다.

## **3\. 실무 사용 예시**

| 상황 | Authority (Server)에서 할 일 | Remote (Client)에서 할 일 |
| :---- | :---- | :---- |
| **데미지 입음** | 캐릭터의 HP 변수 감소, 사망 판정 | (보통 사용 안 함) |
| **아이템 습득** | 인벤토리 배열에 아이템 추가, 필드 아이템 제거 | 습득 효과음 재생, UI 알림 표시 |
| **게임 시작** | 게임 모드 설정, 스폰 지점 결정 | 로딩 화면 해제, HUD 생성 |

## **4\. 주의사항**

1. **Actor Role의 이해:** 이 노드는 내부적으로 액터의 Local Role 변수를 확인합니다.  
   * 서버에서 이 노드를 호출하면 항상 Authority로 흐릅니다.  
   * 클라이언트에서 이 노드를 호출하면 해당 액터가 본인 소유든 아니든 보통 Remote로 흐릅니다.  
2. **함수 호출 방식과의 조합:** Custom Event의 복제 설정(Run on Server, Multicast 등)과 함께 사용할 때 가장 강력합니다. 서버 전용 이벤트 내부에서 다시 한번 Authority 체크를 하는 것은 일종의 '이중 잠금' 역할을 하기도 합니다.

## **5\. 요약**

Switch has Authority는 \*\*"이 일을 서버가 처리할 것인가, 아니면 각자의 화면(클라이언트)에서 처리할 것인가?"\*\*를 결정하는 게이트키퍼입니다. 멀티플레이어 로직의 시작점에는 항상 이 노드가 있다고 생각하면 됩니다.
