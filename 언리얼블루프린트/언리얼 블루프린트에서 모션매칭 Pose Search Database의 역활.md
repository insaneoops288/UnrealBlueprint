# **포즈 서치 데이터베이스 (Pose Search Database)의 역할**

모션 매칭 시스템에서 \*\*포즈 서치 데이터베이스(PSD)\*\*는 애니메이션 에셋들을 분석하여 검색 가능한 형태의 '데이터 세트'로 변환하고 관리하는 역할을 수행합니다.

## **1\. 핵심 역할: 애니메이션 데이터의 인덱싱 (Indexing)**

PSD는 단순히 애니메이션을 담아두는 바구니가 아니라, 애니메이션의 모든 프레임을 쪼개서 수치화(Indexing)합니다.

* **특징 추출**: 연결된 \*\*포즈 서치 스키마(Pose Search Schema)\*\*를 기반으로 각 프레임의 본(Bone) 위치, 속도, 캐릭터의 이동 궤적(Trajectory) 등을 데이터화합니다.  
* **검색 최적화**: 수만 개의 포즈 중에서 실시간(매 프레임)으로 가장 유사한 것을 찾아야 하므로, 내부적으로 KD-Tree와 같은 고속 검색 구조를 생성하여 성능을 최적화합니다.

## **2\. 런타임 쿼리 처리 (Query & Search)**

애니메이션 블루프린트(ABP)의 **Motion Matching 노드**가 현재 캐릭터의 상태(입력값, 현재 포즈 등)를 PSD에 질문(Query)하면, PSD는 답변을 내놓습니다.

* **비용 계산 (Cost Calculation)**: 캐릭터가 가고자 하는 방향(궤적)과 현재 포즈가 데이터베이스 내의 포즈들과 얼마나 차이가 나는지 '비용(Cost)'을 계산합니다.  
* **최적 포즈 선택**: 비용이 가장 낮은, 즉 현재 상황과 가장 유사하고 자연스럽게 연결될 수 있는 포즈를 선택하여 즉시 재생합니다.

## **3\. 스테이트 머신 지옥 (State Machine Hell) 탈출**

기존 방식과의 결정적인 차이점입니다.

* **로직 자동화**: "걷기에서 달리기로 전환할 때 어떤 블렌딩을 쓸 것인가?"라는 규칙을 일일이 블루프린트로 짤 필요가 없습니다.  
* **데이터 기반**: PSD에 적절한 애니메이션 데이터(Sequence, BlendSpace 등)를 추가하기만 하면, 시스템이 알아서 최적의 전환 지점을 찾아냅니다.

## **4\. 구성 요소와의 관계**

* **Pose Search Schema**: PSD가 "무엇을 기준으로 검색할 것인가(발의 위치인가, 몸의 속도인가?)"를 정의하는 설계도입니다. PSD는 이 설계도에 맞춰 데이터를 가공합니다.  
* **Character Trajectory Component**: 캐릭터가 앞으로 이동할 예상 경로를 제공합니다. PSD는 이 경로와 가장 잘 맞는 애니메이션을 DB에서 찾습니다.  
* **Chooser (선택 사항)**: 상황(전투 중, 평상시 등)에 따라 어떤 PSD를 사용할지 결정하는 상위 레이어 역할을 합니다.

## **요약**

| 구분 | 역할 및 특징 |
| :---- | :---- |
| **정적 역할** | 애니메이션 프레임별 특징(위치, 속도, 궤적)을 수치화하여 저장 |
| **동적 역할** | 현재 캐릭터의 입력/상태와 가장 '비용'이 적은 포즈를 실시간 검색 |
| **활용 장점** | 애니메이터의 수동 트랜지션 작업 최소화, 극도로 자연스러운 모션 연결 |

**Tip:** PSD 내에서 특정 애니메이션의 가중치를 조절하거나, 특정 본의 검색 중요도를 높여 캐릭터의 움직임 특성(예: 발의 정확도 중시 등)을 세밀하게 조정할 수 있습니다.
