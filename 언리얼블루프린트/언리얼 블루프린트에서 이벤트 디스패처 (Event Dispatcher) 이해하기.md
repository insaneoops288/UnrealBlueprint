# **언리얼 블루프린트: 이벤트 디스패처 (Event Dispatcher) 이해하기**

이벤트 디스패처는 블루프린트 간의 통신을 **느슨한 결합(Loose Coupling)** 상태로 유지하면서 효율적으로 데이터를 전달하거나 신호를 보낼 때 사용합니다.

## **1\. 이벤트 디스패처의 핵심 개념**

이벤트 디스패처는 **발행-구독(Publish-Subscribe) 모델**을 따릅니다.

* **Call (호출):** 신호를 보내는 행위 (방송하기).  
* **Bind (바인드):** 신호를 기다리다가 신호가 오면 실행할 함수를 연결하는 행위 (구독하기).  
* **Unbind (언바인드):** 더 이상 신호를 듣지 않겠다고 연결을 끊는 행위.

## **2\. 'Call' 노드의 역할**

이벤트 디스패처를 생성한 후 그래프로 드래그하여 Call을 선택하면 호출 노드가 생성됩니다.

### **동작 원리**

1. **Call** 노드가 실행되면, 해당 디스패처에 **Bind**된 모든 이벤트나 함수들이 즉시 실행됩니다.  
2. 만약 바인드된 대상이 아무도 없다면, **Call**을 해도 아무런 일도 일어나지 않습니다 (오류가 발생하지도 않습니다).  
3. 매개변수(Inputs)를 설정했다면, **Call** 할 때 보낸 데이터가 모든 구독자에게 동일하게 전달됩니다.

## **3\. 실전 예제: 버튼과 전등**

가장 흔한 예시는 "스위치를 눌렀을 때 여러 개의 전등이 동시에 켜지는 상황"입니다.

### **\[발행자: BP\_Switch\]**

1. **My Dispatchers** 패널에서 OnSwitchPressed라는 디스패처를 생성합니다.  
2. 스위치 작동 로직 끝에 Call OnSwitchPressed 노드를 연결합니다.

### **\[구독자: BP\_Light\]**

1. **BeginPlay**에서 스위치(BP\_Switch) 참조 변수를 가져옵니다.  
2. 스위치 변수에서 Bind Event to OnSwitchPressed 노드를 실행합니다.  
3. 이벤트 핀에 실제 전등을 켜는 커스텀 이벤트를 연결합니다.

## **4\. 왜 인터페이스 대신 디스패처를 쓰나요?**

| 특징 | 인터페이스 (Interface) | 이벤트 디스패처 (Event Dispatcher) |
| :---- | :---- | :---- |
| **방향성** | 1대1 통신에 가까움 (특정 대상에게 명령) | 1대N 통신 (방송 시스템) |
| **의존성** | 호출자가 대상을 알아야 함 | 호출자는 누가 듣는지 몰라도 됨 |
| **활용** | 상호작용(문 열기, 아이템 줍기 등) | 상태 변화 알림(체력 0, 게임 시작, 폭발 등) |

## **5\. 주의사항 및 팁**

* **유효성 검사 (Is Valid):** Call을 하기 위해 다른 액터의 디스패처를 참조할 때는 항상 그 액터가 유효한지 체크하는 것이 좋습니다.  
* **메모리 관리:** BeginPlay에서 바인드했다면, 액터가 파괴될 때나 필요 없을 때 Unbind를 해주는 습관을 들이면 좋습니다. (대부분의 경우 언리얼이 자동으로 처리하지만 복잡한 시스템에서는 중요합니다.)  
* **매개변수:** 디스패처의 'Details' 패널에서 입력을 추가하면, Call 할 때 정보를 실어 보낼 수 있습니다. (예: 누가 대미지를 입혔는지, 얼마만큼 입혔는지 등)
