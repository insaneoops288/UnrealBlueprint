# **언리얼 엔진 블루프린트: 함수 vs 매크로 가이드**

언리얼 엔진 블루프린트에서 로직의 재사용성을 높이기 위해 사용하는 \*\*함수(Function)\*\*와 \*\*매크로(Macro)\*\*의 핵심 차이점을 분석합니다.

## **1\. 근본적인 차이: 호출 vs 인라이닝**

가장 중요한 기술적 차이는 \*\*"컴파일 시점에 로직이 어떻게 배치되는가"\*\*에 있습니다.

### **함수 (Function)**

* **호출 방식 (Call):** 함수는 메모리 상에 한 번만 존재합니다. 실행 시점에 해당 함수가 있는 메모리 주소로 점프하여 로직을 수행한 후 다시 돌아옵니다.  
* **특징:** 메모리 효율이 높고, 복잡한 로직을 반복 호출할 때 유리합니다.

### **매크로 (Macro)**

* **인라이닝 방식 (Inlining):** 컴파일 단계에서 매크로가 사용된 모든 위치에 매크로 내부의 노드들을 그대로 **'복사해서 붙여넣기'** 합니다.  
* **특징:** 실행 시점의 점프 비용이 없지만, 아주 큰 매크로를 수백 번 사용하면 컴파일된 블루프린트의 크기가 비대해질 수 있습니다.

## **2\. 주요 기능 비교 매트릭스**

| 기능 | 함수 (Function) | 매크로 (Macro) |
| :---- | :---- | :---- |
| **실행 핀 (Exec)** | 입력 1개, 출력 1개로 고정 | **여러 개의 입/출력 가능** (예: Success/Failed 분기) |
| **지연 노드 (Latent)** | Delay, Timeline 등 사용 불가 | **사용 가능** |
| **지역 변수 (Local Var)** | **지원함** (함수 내에서만 쓰는 임시 변수) | 지원 안 함 |
| **상속 및 오버라이딩** | **가능** (자식 클래스에서 재정의 가능) | 불가능 |
| **순수(Pure) 노드** | 지원함 (실행 핀 없는 계산 전용 노드) | 지원 안 함 |
| **데이터 타입** | 명확한 타입 지정 필요 | **Wildcard** 지원 (다양한 타입을 유연하게 처리) |

## **3\. 함수(Function)의 상세 특징**

### **장점**

1. **재귀 호출:** 자기 자신을 다시 호출하는 로직을 구현할 수 있습니다.  
2. **캡슐화:** 지역 변수를 통해 내부 로직에 필요한 임시 데이터를 깔끔하게 관리할 수 있습니다.  
3. **상속 활용:** 부모 클래스에서 정의한 로직을 자식 클래스에서 Override하여 다르게 작동하게 만들 수 있습니다.

### **단점**

* 시간의 흐름을 제어하는 노드(Delay, Timeline, Wait for...)를 내부에 넣을 수 없습니다. 이는 함수가 호출된 프레임 안에 즉시 완료되어야 한다는 원칙 때문입니다.

## **4\. 매크로(Macro)의 상세 특징**

### **장점**

1. **흐름 제어 최적화:** Branch나 Switch를 포함하여 실행 경로를 여러 갈래로 나누어 출력할 수 있습니다. (예: DoOnce, FlipFlop 같은 노드들은 모두 매크로입니다.)  
2. **비동기 작업:** Delay를 포함할 수 있어, 일정 시간 뒤에 실행되어야 하는 일련의 과정을 묶기에 적합합니다.  
3. **와일드카드:** 입력 타입을 특정하지 않고 들어오는 값에 따라 유연하게 대응하는 노드를 만들 수 있습니다.

### **단점**

* 내부 변수를 저장할 수 없어 상태 관리가 어렵습니다.  
* 클래스 간 상속 구조에서 다형성을 활용하기 어렵습니다.

## **5\. 상황별 선택 기준 (Best Practices)**

### **언제 함수를 쓰나요?**

* 수학적 계산이나 특정 값을 반환하는 로직일 때.  
* 다른 블루프린트에서도 호출해야 하는 공용 인터페이스일 때.  
* 클래스 상속 구조를 활용하여 기능을 확장해야 할 때.  
* 내부에서 임시로 데이터를 저장하고 계산해야 할 때.

### **언제 매크로를 쓰나요?**

* Delay나 Timeline이 포함된 로직 덩어리를 정리할 때.  
* 실행 결과에 따라 Found, Not Found 처럼 출력 핀을 여러 개로 쪼개고 싶을 때.  
* 노드 그래프가 너무 복잡해서 단순히 '시각적으로' 깔끔하게 묶고 싶을 때.  
* 여러 데이터 타입을 동시에 처리하는 범용적인 유틸리티 노드를 만들 때.

### **요약**

"기능적으로 복잡하고 상속이 필요하면 **함수**, 흐름 제어(Delay/Multi-Exec)가 핵심이고 시각적인 정리가 목적이면 **매크로**를 선택하세요."
